# Elasticsearch, Kibana Discover, KQL, Tables, and Visualizations

## Overview

This repository contains my learning notes on Elasticsearch and Kibana, with a focus on:

* Searching logs in Kibana Discover
* Using Kibana Query Language (KQL)
* Viewing data in tables
* Creating visualizations
* Investigating security and operational logs

Elasticsearch stores and indexes data, while Kibana provides a web interface to search, analyze, visualize, and investigate that data.

---

## What Is Elasticsearch?

Elasticsearch is a distributed search and analytics engine used to store, search, and analyze large volumes of data.

It is commonly used for:

* Application logs
* Windows event logs
* Linux system logs
* Firewall logs
* Web server logs
* Network logs
* Security alerts
* Metrics
* Audit logs

Elasticsearch stores data as documents inside indexes.

---

## What Is Kibana?

Kibana is the visualization and analysis interface used with Elasticsearch.

It allows users to:

* Search logs
* Filter events
* Investigate alerts
* Create tables
* Build dashboards
* Create charts and graphs
* Monitor systems
* Analyze security events

---

## Elasticsearch and Kibana Workflow

A common workflow looks like this:

```text
Log Source → Elastic Agent / Beats → Elasticsearch → Kibana
```

Example:

```text
Windows Event Logs → Elastic Agent → Elasticsearch Index → Kibana Discover
```

The log source sends data to Elasticsearch, and Kibana helps analysts search and visualize that data.

## Important Terms

### Document

A document is one record or event stored in Elasticsearch.

Example:

```json
{
  "@timestamp": "2026-07-04T10:30:00Z",
  "user": {
    "name": "admin"
  },
  "source": {
    "ip": "192.168.1.10"
  },
  "event": {
    "action": "login_failed"
  }
}
```

Each log event can be stored as a document.

---

### Index

An index is a collection of related documents.

Examples:

* `logs-windows-*`
* `logs-system-*`
* `logs-firewall-*`
* `logs-nginx-*`
* `logs-endpoint.events-*`

An index is similar to a database table, but it is designed for fast searching and analytics.

---

### Field

A field is a named piece of information inside a document.

Examples:

* `@timestamp`
* `user.name`
* `source.ip`
* `host.name`
* `event.action`
* `process.name`
* `message`

---

### Data View

A data view tells Kibana which indexes to use.

Example data view:

```text
logs-*
```

This data view can match multiple indexes that start with `logs-`.

## Kibana Discover

Discover is the Kibana section used to search and investigate raw log events.

It is commonly used by SOC analysts, DevOps engineers, and system administrators.

In Discover, users can:

* View incoming log events
* Search logs using KQL
* Filter data by fields
* Change the time range
* Add columns to the event table
* Inspect individual events
* Save searches
* Export results

---

## Discover Layout

The Discover page usually contains:

* **Data View Selector**: Selects the indexes to search.
* **Time Filter**: Limits results to a selected time range.
* **KQL Search Bar**: Used to search and filter events.
* **Field List**: Displays available fields in the selected data view.
* **Event Table**: Displays matching log events.
* **Document Details Panel**: Shows complete details of a selected event.

---

## Using the Time Filter

The time filter is important because Elasticsearch may contain millions of events.

Common time ranges include:

* Last 15 minutes
* Last 1 hour
* Last 24 hours
* Last 7 days
* Custom range

If no results appear, check the time range first.

For example, logs may exist in the index, but not within the currently selected time range.

## What Is KQL?

KQL stands for **Kibana Query Language**.

KQL is used in Kibana to search and filter data.

KQL helps analysts find specific events based on fields and values.

KQL is mainly used for filtering data. It is not the same as Elasticsearch Query DSL.

---

## Basic KQL Syntax

### Search for a Word

```kql
failed
```

This searches for the word `failed` in searchable fields.

---

### Search a Specific Field

```kql
user.name: admin
```

This finds events where the username is `admin`.

---

### Search an IP Address

```kql
source.ip: 192.168.1.10
```

This finds events from the source IP address `192.168.1.10`.

---

### Search a Hostname

```kql
host.name: WIN-CLIENT-01
```

This finds events generated by the specified host.

---

### Search a Process Name

```kql
process.name: powershell.exe
```

This finds events related to PowerShell.

---

### Search an Event Action

```kql
event.action: login_failed
```

This finds failed login events.

---

## KQL Operators

### AND

Use `and` when all conditions must match.

```kql
event.action: login_failed and user.name: admin
```

This finds failed logins for the `admin` account.

---

### OR

Use `or` when either condition can match.

```kql
process.name: powershell.exe or process.name: cmd.exe
```

This finds PowerShell or Command Prompt activity.

---

### NOT

Use `not` to exclude events.

```kql
not user.name: SYSTEM
```

This excludes events created by the `SYSTEM` account.

---

### Parentheses

Use parentheses to group conditions.

```kql
(event.action: login_failed or event.action: login_success) and user.name: admin
```

This finds successful or failed login events for the `admin` account.

---

## Wildcards in KQL

Wildcards can match partial values.

```kql
user.name: adm*
```

This can match usernames such as:

* `admin`
* `administrator`
* `admin01`

Another example:

```kql
process.name: *powershell*
```

This can match values containing `powershell`.

---

## Useful KQL Examples for Security Analysis

### Find Failed Login Events

```kql
event.action: login_failed
```

---

### Find Failed Logins From One IP Address

```kql
event.action: login_failed and source.ip: 192.168.1.10
```

---

### Find PowerShell Activity

```kql
process.name: powershell.exe
```

---

### Find Command Prompt Activity

```kql
process.name: cmd.exe
```

---

### Find Events From a Specific Host

```kql
host.name: WIN-CLIENT-01
```

---

### Find Events From a Specific User

```kql
user.name: john
```

---

### Find Events Outside Business Hours

```kql
@timestamp >= "2026-07-04T00:00:00Z" and @timestamp < "2026-07-04T08:00:00Z"
```

This can help identify activity outside expected working hours.

---

### Find Suspicious Script Activity

```kql
process.name: powershell.exe and process.command_line: *encodedcommand*
```

This can help identify PowerShell commands using encoded command arguments.

## Tables in Kibana Discover

The event table in Discover displays matching log events.

By default, it may show fields such as:

* Time
* Document
* Message

You can customize the table by adding fields as columns.

Useful fields for security investigations include:

* `@timestamp`
* `host.name`
* `user.name`
* `source.ip`
* `destination.ip`
* `event.action`
* `event.category`
* `process.name`
* `process.command_line`
* `file.name`
* `file.hash.sha256`
* `message`

---

## How to Create a Table in Discover

1. Open **Kibana Discover**.
2. Select the correct data view.
3. Set the correct time range.
4. Enter a KQL query.
5. Search for a field in the field list.
6. Click the **Add** button beside the field.
7. The selected field appears as a column in the event table.
8. Repeat the process for other useful fields.
9. Save the search if needed.

Example table columns for failed-login investigation:

| Time         | User        | Source IP   | Host        | Event Action   |
| ------------ | ----------- | ----------- | ----------- | -------------- |
| `@timestamp` | `user.name` | `source.ip` | `host.name` | `event.action` |

---

## Sorting and Filtering Tables

In Discover, analysts can:

* Sort a column in ascending or descending order.
* Click a field value to filter for that value.
* Exclude a field value from results.
* Add fields as columns.
* Remove unnecessary columns.
* Open a document to view complete event details.

Example investigation flow:

1. Search for failed login events.
2. Add `user.name`, `source.ip`, and `host.name` as columns.
3. Sort by time.
4. Check whether the same source IP attempted many logins.
5. Investigate whether a successful login occurred after repeated failures.

## Lens

Lens is a Kibana feature used to create visualizations using a drag-and-drop interface.

It is useful for building charts and tables without writing complex queries.

With Lens, users can:

* Drag fields into the workspace.
* Select a chart type.
* Count events.
* Group events by field.
* Filter results using KQL.
* Create time-based charts.
* Save visualizations to dashboards.

---

## Creating a Data Table Visualization

Example: Create a table of failed login attempts by username.

1. Open **Analytics → Dashboard** or **Visualize Library**.
2. Create a new visualization.
3. Select **Lens**.
4. Select the required data view.
5. Add a KQL filter:

```kql
event.action: login_failed
```

6. Set the metric to **Count of records**.
7. Add a breakdown or row field such as `user.name`.
8. Choose the **Data Table** visualization type.
9. Sort by count in descending order.
10. Save the visualization.

Example output:

| User     | Failed Login Count |
| -------- | -----------------: |
| admin    |                 45 |
| john     |                 18 |
| testuser |                  7 |

## Detecting Anomalies With Kibana

An anomaly is activity that is unusual compared to normal behavior.

Kibana visualizations can help identify anomalies by showing spikes, rare values, and unusual trends.

Examples include:

* A sudden spike in failed login attempts
* One IP address connecting to many hosts
* A user logging in at unusual hours
* A new process appearing on many endpoints
* A sudden increase in PowerShell activity
* A large increase in outbound network traffic
* An endpoint generating unusually high event volume

Example approach:

1. Create a line chart for failed logins over time.
2. Identify unusual spikes.
3. Filter the spike time range in Discover.
4. Add useful fields such as user, source IP, host, and event action.
5. Investigate the related events.

## SOC Investigation Workflow in Kibana

1. Receive an alert from a SIEM, EDR, or detection rule.
2. Open Discover.
3. Select the correct data view.
4. Set the correct time range.
5. Search using KQL.
6. Add relevant fields to the event table.
7. Inspect individual documents.
8. Identify affected users, hosts, processes, IP addresses, and files.
9. Use visualizations to identify trends or spikes.
10. Document findings and escalate or close the alert.
