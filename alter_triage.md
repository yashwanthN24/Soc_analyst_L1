# Alert Triage

Alert triage is the process of **reviewing, validating, prioritizing, and investigating security alerts** generated by security tools such as SIEM, EDR, IDS/IPS, firewalls, email security gateways, and cloud security platforms.

The primary objective of alert triage is to determine:

- Is the alert legitimate?
- Does it indicate malicious activity?
- What is the impact?
- What actions should be taken?

SOC Level 1 analysts spend most of their day performing alert triage.

---

# Why is Alert Triage Important?

Modern organizations generate thousands or even millions of alerts every day.

Not every alert represents an attack.

Some alerts are:

- False positives
- Misconfigurations
- Normal administrative activity
- Legitimate software behavior

Without triage, analysts would waste valuable time investigating harmless events while real attacks remain unnoticed.

Alert triage helps analysts:

- Reduce alert fatigue
- Prioritize critical incidents
- Detect attacks early
- Improve incident response efficiency
- Prevent business impact

---

# Alert Triage Workflow

```
Alert Generated
       │
       ▼
Collect Context
       │
       ▼
Validate the Alert
       │
       ▼
Determine Severity
       │
       ▼
Investigate
       │
       ▼
Decision
       │
       ├── Close
       ├── Monitor
       └── Escalate
```

---

# Phase 1 — Alert Reception

## What happens?

A security tool detects suspicious behavior and generates an alert.

Examples:

- Multiple failed logins
- Malware detection
- PowerShell execution
- Impossible travel login
- Privilege escalation
- Suspicious network connection

---

## Why is this phase needed?

Without alerts, analysts would have no indication that suspicious activity occurred.

Security tools continuously monitor the environment and notify analysts whenever predefined detection rules are triggered.

---

# Phase 2 — Collect Context

## What happens?

Gather all available information related to the alert.

Typical information includes:

- Username
- Hostname
- IP Address
- Process Name
- Command Line
- Timestamp
- Parent Process
- Destination IP
- File Hash
- Alert Rule
- MITRE ATT&CK Technique
- Previous alerts

---

## Why is this phase needed?

A single alert rarely provides enough information.

Additional context helps determine whether the activity is normal or malicious.

---

# Phase 3 — Validate the Alert

## What happens?

Determine whether the alert actually represents suspicious activity.

Ask questions such as:

- Did the activity really occur?
- Is the detection rule functioning correctly?
- Is this expected administrator activity?
- Is it a scheduled task?
- Is it approved software?
- Is the behavior common in the environment?

---

## Possible Outcomes

### True Positive

Malicious activity actually occurred.

Example:

PowerShell downloading malware from the Internet.

---

### False Positive

Legitimate activity incorrectly identified as malicious.

Example:

An administrator legitimately runs PowerShell maintenance scripts.

---

### Benign Positive

The activity occurred but is expected and authorized.

Example:

IT team creating new Active Directory users.

---

## Why is this phase needed?

Investigating every alert wastes analyst time.

Validation filters out unnecessary investigations.

---

# Phase 4 — Determine Severity

## What happens?

Estimate the potential impact if the alert is malicious.

Common severity levels:

- Informational
- Low
- Medium
- High
- Critical

---

## Factors considered

- Business impact
- Sensitive assets involved
- Number of affected systems
- Privilege level
- Type of attack
- Ease of exploitation
- Data sensitivity

---

## Why is this phase needed?

Analysts cannot investigate every alert immediately.

Severity determines which alerts receive immediate attention.

---

# Phase 5 — Investigation

## What happens?

Perform a deeper analysis.

Investigate:

- Related logs
- Parent and child processes
- Network connections
- Authentication events
- Registry changes
- File activity
- User behavior
- Endpoint telemetry
- Previous alerts
- Threat intelligence

---

## Questions to Answer

- What happened?
- When did it happen?
- Who performed the activity?
- Which systems are affected?
- Is the attack still active?
- What is the attacker's objective?

---

## Why is this phase needed?

The investigation determines whether an actual security incident exists and identifies its scope.

---

# Phase 6 — Decision

Based on the investigation, decide the next action.

---

## Close the Alert

When:

- False positive
- Benign activity
- Duplicate alert
- Expected behavior

---

## Continue Monitoring

When:

- Suspicious but inconclusive
- Additional evidence may appear
- Low risk
- Requires observation

---

## Escalate

When:

- Confirmed malicious activity
- Business impact exists
- Multiple systems affected
- Requires containment
- Requires Incident Response team

SOC L1 usually escalates confirmed incidents to SOC L2 or the Incident Response team.

---

# Escalation Information

A good escalation should include:

- Alert summary
- Timeline
- Affected host
- Username
- IP addresses
- Severity
- Investigation performed
- Evidence collected
- Recommendation

---

# Example

## Alert

```
PowerShell executed with EncodedCommand.
```

### Collect Context

- User: John
- Device: FINANCE-PC01
- Parent Process: Outlook
- Destination IP: Unknown
- Time: 10:15 AM

---

### Validate

PowerShell should not normally launch from Outlook.

Potential phishing attack.

---

### Severity

High

Reason:

- Finance device
- Suspicious PowerShell
- Possible malware execution

---

### Investigation

Findings:

- Email attachment opened
- PowerShell downloaded executable
- Outbound connection established
- Defender detected malware

---

### Decision

✅ Escalate immediately.

---

# Best Practices

- Never assume an alert is malicious without evidence.
- Always gather sufficient context.
- Verify before escalating.
- Follow documented playbooks.
- Record every investigation step.
- Prioritize based on business impact.
- Escalate with complete evidence.
- Maintain clear documentation.

---

# Key Takeaways

- Alert triage is the core responsibility of a SOC Level 1 analyst.
- The goal is to quickly determine whether an alert is real, its severity, and the appropriate response.
- Effective triage reduces false positives, improves response time, and ensures critical incidents are handled promptly.
